# Правила работы агентов

Этот файл обязателен для всех агентов, которые меняют репозиторий RelayApp.

## Коммит после каждой завершённой задачи

- Владелец задачи обязан создать один атомарный коммит после каждой завершённой и проверенной задачи, если она существенно изменила файлы проекта.
- Чтение, исследование, пустой результат, незавершённая работа и промежуточные попытки не создают коммитов.
- В общей рабочей директории коммитит только корневой владелец задачи. Субагенты передают ему изменения и результаты проверок и не выполняют `git commit`, если это не поручено явно.
- В коммит попадают только относящиеся к задаче файлы или отдельные фрагменты изменений. Перед коммитом обязательны `git diff --cached`, проверка отсутствия секретов и релевантные тесты.
- Push, публикация релиза и другие внешние действия не следуют из факта коммита и требуют отдельного разрешения пользователя.
- Пустые коммиты, микрокоммиты на каждое действие инструмента и коммиты заведомо сломанного состояния запрещены.

## Формат сообщения коммита

Заголовок следует Conventional Commits и остаётся техническим. Каждый коммит обязан содержать блок для пользователей:

```text
fix(chat): стабилизировать загрузку истории

Краткое техническое объяснение причины и решения.

Patch-Note:
- Чат теперь сразу открывается у последнего сообщения.
- Поздняя загрузка изображений больше не сдвигает прокрутку.

Verification:
- npm run typecheck
- npm run build
```

Правила блока `Patch-Note`:

- от 1 до 5 коротких, понятных пользователю пунктов на русском языке;
- основная фраза должна быть русской; названия продуктов и форматов вроде RelayApp, WebP или LiveKit допустимы, но английский технический текст с одной русской вставкой проверку не проходит;
- описывать результат, а не файлы, классы, внутренние сервисы или процесс разработки;
- не включать пути, URL, IP, SHA, логи, секреты, токены, ключи, пароли и другие внутренние данные;
- не копировать весь body коммита: в чат публикуется только этот явно отмеченный блок;
- `Patch-Note: skip` разрешён только для изменений без поставляемого поведения: документации, тестов, правил агентов и CI-инфраструктуры. Код приложения, конфигурация рантайма и исправления всегда получают пользовательский патчноут.

`Verification` перечисляет реально выполненные проверки. Если проверку выполнить нельзя, это указывается явно вместо выдуманного результата.

## Как патчноут попадает в чат

- CI собирает пункты `Patch-Note` из всех коммитов текущего push в порядке от старых к новым и формирует один патчноут обновления.
- Вместе с текущим патчноутом CI передаёт ограниченное окно проверенной first-parent истории. Сервер отмечает уже покрытые коммиты и при следующем совместимом успешном релизе переносит вперёд изменения из предыдущего упавшего запуска — без повторов и без потери порядка.
- Патчноут публикуется только после успешного обновления продакшена. Если один push меняет и веб/сервер, и desktop, публикация ждёт успешного завершения всех затронутых компонентов.
- Для одного SHA он публикуется не более одного раза на сервер благодаря серверной идемпотентности.
- Непосредственно перед фактической заменой релиза фиксируется список серверов, где пользователи действительно видят панель чата. Сообщение получают только эти серверы. Если открытых чатов нет, обновление помечается пропущенным и не отправляется позднее.
- Повтор той же попытки не переснимает аудиторию. Новая попытка после неудачного обновления может сделать свежий снимок, только пока ни один обязательный компонент ещё не завершился.
- Если старая версия сервера ещё не умеет сделать точный снимок до обновления, первый rollout механизма честно пропускает патчноут и не восстанавливает аудиторию задним числом.
- Коммиты с `Patch-Note: skip` не создают сообщение в чате.

## Обязательная проверка

- Выполнить `npm run hooks:install` один раз после клонирования. `npm install` в корне также включает hooks автоматически.
- Отслеживаемый `.githooks/commit-msg` запускает тот же валидатор до создания коммита и отклоняет неверный формат, нерусский или небезопасный патчноут и незаконный `Patch-Note: skip`; `.gitmessage` подставляет правильную структуру заранее.
- Workflow `commit-policy` повторяет проверку для push и pull request доверенной версией валидатора из предыдущего защищённого коммита. Deploy/build также запускают проверку до обновления, поэтому обычный обход локального hook не выпускает некорректный патчноут.
- В GitHub ветка `main` должна быть защищена: только pull request, обязательный успешный `commit-policy`, запрет force push и обязательное одобрение CODEOWNER для файлов политики из `.github/CODEOWNERS`. Локальный hook и CODEOWNERS сами по себе не являются непреодолимой защитой без этих настроек репозитория.
