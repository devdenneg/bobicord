## Что чинить (по приоритету)

1. **IDR-шторм** — главный источник фризов «без потерь» на всех стримах. Разобраться, почему `kfBackoffMs`-бэкофф в `onRequestKeyframe` не гасит частоту до ожидаемых 0.25/с. IDR ~1/с при 1080p — это ~200–300 КБ залпом каждую секунду.
2. **GOP транскода под 60fps** — `-g 60` в `apps/relay-core/src/transcode.rs:390` захардкожен под 30fps; при 60fps-пресетах рендишн печёт keyframe раз в 1с. Фикс: `GOP = 2 × fps`.
3. **60fps перегружает вещателя** — бюджет кадра падает вдвое (33→16.7мс), CPU-путь захвата не успевает (cb до 72мс). Либо адаптация к CPU, либо дефолт 30fps.
4. **Атомарность переезда coturn** — клиенты держат старые ICE-серверы; при смене медиа-хоста нужен форс-рефреш `welcome`/переподключение.

## Разбор IDR-шторма (дополнено 2026-07-11)

Разведка кода всех путей, которыми может испечься IDR, объясняет несоответствие **PLI=5 vs IDR=2270 за ~2270с**:

| Путь | Лимит | Мог дать 1/с? |
|---|---|---|
| Сервер: проброс `request-keyframe` (`tree.js:1564`, `onRequestKeyframe`) | бэкофф 1→8с на дерево | нет (PLI=5) |
| vrelay: gap-heal / PLI детей / Connected (`relay.rs`) | общий `last_kf_ms` ≥1с | нет |
| Вещатель: RTCP PLI от детей (`peer.rs:133-155`) | `last_pli_ms` 1с, счётчик PLI считает ДО лимита | нет (PLI=5) |
| ffmpeg-рендишн (`transcode.rs`, `-g 60`) | свои IDR, к source-вещателю НЕ уходят | нет |
| **Энкодер вещателя, периодический GOP** | задуман fps×4 (0.25/с); **дефолт HW MFT = GOP=fps = 1/с** | **ДА — источник** |

**Вывод: IDR-шторм — не петля PLI, а энкодер на дефолтном GOP=fps.** Все запросные пути зарейтлимичены и при PLI=5 молчали; единственный некапнутый периодический источник — сам аппаратный MFT, у которого не применился GOP. Серверный `kfBackoffMs` «не гасил частоту», потому что гасить нечего: IDR генерились таймером внутри энкодера, до которого бэкофф не дотягивается.

**Фикс уже закоммичен**: `49b4083` (2026-07-10 17:24 МСК) — `MF_MT_MAX_KEYFRAME_SPACING` ставится на выходном типе ДО `SetOutputType` (`encoder.rs:82-85`). Вечерние стримы 10.07 (и Франкфурт-, и Москва-отрезки) шли на бинаре БЕЗ этого фикса — диаг подтверждает старое поведение, а не опровергает фикс.

**Верификация тест-стримом** (следующий шаг):
1. Вещатель обновляется до бинаря ≥ `49b4083`, стримит несколько минут.
2. `npm run diag -- pull --stream=<id>` + `report` — ожидание: `IDR ≈ 0.25/с` (GOP=4с) при `PLI ≈ 0`, фризы «без потерь» у зрителей уходят.

**Contingency, если IDR остался ~1/с на новом бинаре**: конкретный HW MFT игнорирует и spacing-атрибут, уважая только `CODECAPI_AVEncMPVGOPSize`, выставленный ДО `SetOutputType` (сейчас он ставится ПОСЛЕ, `encoder.rs:115`). Тогда:
- переставить ICodecAPI-вызов ДО `SetOutputType`;
- залогировать фактический keyframe-spacing (`GetOutputCurrentType` → чтение `MF_MT_MAX_KEYFRAME_SPACING`) сразу после `SetOutputType`;
- в `net:`-строке разделить счётчик IDR на «форс» (по `force_keyframe`) и «периодические» — чтобы шторм был виден сразу, без сведения диага.

## План остальных фиксов (решения 2026-07-11)

1. **GOP транскода → кап рендишнов 30fps** (вместо прокидки fps по цепочке — её нигде нет: натив join → tree.js → vrelay-агент → relay-core, 7 точек). В `ffmpeg_args` (`transcode.rs:372`) добавить `fps=30,` ПЕРВЫМ звеном `-vf` (дроп кадров до scale — минус CPU VPS; честный CFR, консистентно с `force-cfr=1`); `-g 60` становится честными 2с при любом исходнике. Сопутствующее: подрезать очередь `in_marks` в reader'е (метка на каждый входной кадр, поп на каждый выходной — при 60→30 метрика латентности деградирует в мусор): `while q.len() > 8 { q.pop_front(); }`.
2. **CPU-адаптация вещателя**: авто-снижение fps 60→30 по CPU-сигналу в stats-тике (`mod.rs:651-665`, cb_avg/cb_n/drops уже меряются). Критерий — утилизация колбэка `cb_avg×cb_n/(dt×1000) > 0.85` ИЛИ `capture_drops > 10%` ожидаемых кадров, 3 тика (6с) подряд; ниже 30 не режем; **без возврата вверх** (latch до конца сессии — по метрикам нельзя отличить «появился запас» от «нагрузка та же», проба вверх = реинит MFT + IDR-спайк + болтанка). Низ готов: смена `targets.fps` → capture подхватывает на след. кадре, энкодер пересоздаёт MFT с новым GOP + IDR (`mod.rs:393`). Анти-конфликт с `QualityLadder`: поле `cpu_fps_cap`, ladder остаётся единственным писателем `targets.fps`. В дебаг-панель — бейдж `(CPU)`.
3. **UI**: добавить 1080p30/720p30 в сегмент качества `BroadcastModal` (значения из `PRESETS`, не дублировать); дефолт «Авто» не меняется.
4. **ICE при переезде медиа-хоста — ops-процедура, без кода** (переезды редки): поднять новый медиа-стек → обновить `TURN_URLS`/`STUN_URLS`/`TURN_EXTERNAL_IP` в прод-`.env` → **рестарт сервера** (рвёт все tree-WS → клиенты реконнектятся → свежий `welcome` с новыми ICE) → только потом гасить старый coturn. Причина: `welcome` шлётся раз на сокет, живые PC не переконфигурируются, версии у списка нет.

5. **UX: смена качества пересоздаёт окно стрима — сделать бесшовной.** Смена = `unwatch()+watch()` (`treeVideo.ts:518-529`): `unwatch` зовёт `delVideo` → стрим уходит из `getStreams()` → `StreamTile` (`ServerView.tsx:1512`) размонтируется → новый `ontrack` монтирует заново. Отсюда чёрный пропад, пересоздание `<video>`, вылет из фуллскрина. Фикс целиком в `treeVideo.ts`, UI не трогается:
   - **Контейнер-`MediaStream` на streamId**, живущий поверх PC: на `ontrack` — `removeTrack` старого трека того же kind + `addTrack` нового (аудио и видео). `<video>.srcObject` остаётся тем же объектом → картинка переключается сама. Критично: эффект `StreamTile` зависит только от `streamKey` (`:1261`), новый handle при том же ключе он не подхватит — поэтому именно подмена треков в одном контейнере, не новый handle.
   - **`unwatch` с флагом `keepVideo`**: в `setQuality` сносить watch без `delVideo` (и в `nativeUnwatch` — у натива тот же путь через единый Rust-слот). Плитка живёт, показывает последний кадр до нового трека.
   - **Failsafe**: 10с-таймер на переключение — трек не пришёл → честный `delVideo` + тост (иначе вечный замороженный кадр). Фолбэк `rendition-unavailable → source` (`engine.ts:232`) идёт тем же бесшовным путём.
   - Бонус тем же механизмом: реконнект watch при обрыве WS (`ws.onclose → teardownWatch → re-watch через 3с`) тоже перестаёт ронять окно.

### Разбор потерь на московском VPS (по логам 77.110.113.200, 11.07)

Ingest-лог vrelay подтверждает: разрывы **на входе** — 90–98% каждую минуту с 11:01 до 11:29 (эскалация 7%→18%→46%→90% за 30 секунд с 11:01:00), и в видео, и в аудио. При этом **тот же artem161 в 11:45 льёт 5.7 Мбит/с с 0.00% разрывов** — состояние бинарное «сломано/чисто».

Проверено и исключено:
- **NIC/IP-стек хоста чист**: `RX dropped 0`, `IpInDiscards 0`, conntrack 130/65536, фрагментация 0.
- **TURN не в тракте**: у coturn ноль аллокаций за оба окна — оба стрима шли прямым UDP.
- **CPU steal**: кумулятивно 0.95% за 18ч аптайма — большого шторма от соседа по гипервизору не было (сейчас фон 4–9%).

Что нашлось:
- **`UdpRcvbufErrors 66138`** (единственный ненулевой счётчик потерь на хосте) — ядро дропало входящий UDP при переполнении приёмного буфера сокета (rmem уже 8 МБ). По времени это коррелирует с **окнами транскода**: бокс **2 vCPU**, ffmpeg-рендишн ест ~100% одного ядра (замер вживую: `rendition=720 cpu=97–104%, latency≈1.3с` — транскод сам не успевает), плюс фанаут vrelay. Худшие окна зрителей (11:24–11:29, loss 35–76%) совпадают с активным 1080p-транскодом.
