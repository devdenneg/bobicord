# Захват звука стрима: «всё кроме RelayApp» через мульти-INCLUDE

Документ описывает фикс захвата аудио для трансляции (нативный вещатель, Tauri/Windows).
Реализация — `apps/native/src-tauri/src/broadcast/audio.rs`.

## Проблема

Стрим должен нести звук игр/системы, но НЕ вывод самого RelayApp (голос из
голосового канала, чужие стримы) — иначе эхо/петля (CLAUDE.md, инвариант 6).

Захват идёт через **WASAPI process loopback**. Фундаментальное ограничение API:
активация берёт **только один PID + дерево процессов**. Отсюда оба прежних режима
хрупкие:

- **EXCLUDE(себя)** — исключить наше дерево процессов из системного микса.
  Промахивается, когда аудио-подпроцесс WebView2 (в нём играет голос войса) не
  оказывается в дереве нашего процесса. На части машин так и было → голос эхом
  уходил в стрим.
- **INCLUDE(pid игры)** — захватить только выбранный процесс. Надёжно режет чужое,
  но даёт **тишину**, когда звук игры родится в соседнем процессе (лаунчер, движок,
  античит-обёртка). Отсюда «работает не на всех машинах».

API на машинах присутствует (INCLUDE же работал) — корень в **топологии процессов**,
не в версии Windows.

## Решение

Режим `ExcludeSelfViaInclude` (новый дефолт) реализует «всё кроме RelayApp»
**по построению**, без зависимости от топологии WebView2 и без выбора процесса
пользователем:

1. Перечислить активные render-аудиосессии на дефолтном устройстве вывода.
2. Вычесть **наши** процессы.
3. Поднять по отдельному **INCLUDE-loopback** клиенту на каждый оставшийся (не-наш)
   PID.
4. **Микшировать** их PCM в один поток → Opus.
5. Раз в ~1с переперечислять: добавлять клиентов для новых источников, убирать
   мёртвых.

`IncludeProcess(pid)` остался как **ручной override** (один процесс, без
перечисления и self-фильтра).

## Архитектура

```
run_capture_loop (супервизор, аудио-поток)
├── раз в ~1с: wanted = list_active_render_pids() − self_pid_set()
│     ├── новый PID  → spawn client-поток
│     └── мёртвый/ушедший PID → stop + join + remove из микса
├── каждые 20мс: Mixer.pull(1920) → Opus.encode → on_chunk
│
└── client-поток (на каждый источник)
      activate_include_loopback(pid) → Initialize → цикл GetBuffer
      → bytes_to_i16_stereo → Mixer.push(pid, ...)
```

### Ключевые функции (`audio.rs`)

- **`list_active_render_pids() -> HashSet<u32>`** — COM-цепочка
  `IMMDeviceEnumerator → GetDefaultAudioEndpoint(eRender, eConsole) →
  IAudioSessionManager2 → IAudioSessionEnumerator`; на каждую сессию берём
  `GetProcessId` при `AudioSessionStateActive`, пропуская `pid == 0`.
- **`self_pid_set() -> HashSet<u32>`** — множество «наших» PID, объединение двух
  эвристик (у каждой поодиночке дыра):
  - **(a)** дерево-потомки нашего процесса — снапшот `CreateToolhelp32Snapshot` +
    транзитивное замыкание по `th32ParentProcessID`. Ловит WebView2-подпроцессы,
    приклеенные к нам как дети.
  - **(b)** любой `msedgewebview2.exe`, чей образ (`QueryFullProcessImageNameW`)
    лежит под папкой нашего exe. Ловит отвязанные/reparented аудио-utility
    процессы, выпавшие из (a).
- **`activate_include_loopback(pid) -> IAudioClient`** — активация process-loopback
  в INCLUDE-режиме (`ActivateAudioInterfaceAsync`, синхронизация через
  `Arc<(Mutex<bool>, Condvar)>` — см. историю бага с HEAP_CORRUPTION в шапке файла).
- **`Mixer`** — по interleaved-стерео i16 очереди на источник.
  - `push` — добавляет PCM, кап очереди ~200мс (клиент с чуть более быстрым
    endpoint-clock не копит задержку — дропаем старейшее).
  - `pull(n)` — сумма фронтов всех очередей в i32, clamp в i16. Очередь короче `n`
    (источник молчал/отстал) вносит вклад лишь на свою длину — остаток тишина.
    Ноль очередей → тишина.
- **`capture_one(pid, mixer, stop)`** — тело клиентского потока; отказ WASAPI
  возвращает Err, поток взводит `dead`, супервизор его reaps.

### Свойства

- **Единый формат** у всех клиентов (IEEE float стерео 48кГц) → микс без
  пересэмплинга и рассинхрона частот.
- **Пейсинг по wall-clock** (абсолютный дедлайн 20мс) → без накопления дрейфа.
- **Отказоустойчивость**: отказ активации/рантайма отдельного клиента логируется
  и **не роняет** весь захват (прежде любая ошибка убивала аудио-поток). Ноль
  источников — трек шлёт тишину, не залипает.
- **Кап** числа клиентов (8) — ограничение ресурсов.
- Каждый клиентский поток — свой COM-апартамент
  (`CoInitializeEx(COINIT_MULTITHREADED)` / `CoUninitialize`).

## Известные ограничения / риски

1. **Только дефолтное устройство вывода** (default console render endpoint). Звук на
   недефолтном устройстве не попадёт — позже расширить циклом по
   `EnumAudioEndpoints(eRender)`.
2. **Shared Evergreen WebView2** (`Program Files\...\EdgeWebView`) не ловится
   эвристикой (b); опора на (a). Полностью отвязанный audio-utility процесс вне
   папки приложения — теоретическая остаточная утечка голоса; проверять на целевых
   машинах.
3. **Дрейф** между N клиентами — снят единым 48кГц форматом и time-sliced
   суммированием; абсолютное frame-выравнивание вводить только при слышимом дрейфе.

## Проверка

- `cargo check` (с `CMAKE_POLICY_VERSION_MINIMUM=3.5` для opus/CMake 4.x) — чисто.
- Runtime (вручную, ≥2 машины):
  1. Войти в голосовой (WebView2 играет голос) → старт трансляции в авто-режиме.
  2. **AC (инвариант 6):** второй зритель НЕ слышит голос войса в стриме — эха нет.
  3. Запустить игру ПОСЛЕ старта стрима → её звук появляется за ~1-2с
     (переперечисление).
  4. Ручной `IncludeProcess` (выбор процесса в UI) → по-прежнему работает.
  5. Закрыть источник во время стрима → нет краша, трек не залипает.
