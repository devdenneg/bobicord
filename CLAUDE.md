
# CLAUDE.md — bobicord → RelayApp

Этот репозиторий (**bobicord**, «Voice») эволюционирует в **RelayApp**: Discord-подобный апп, где видео-раздача экрана переезжает с LiveKit SFU на **сервер-первую P2P relay-архитектуру**, а голос/чат/auth/деплой остаются как есть. Главный рабочий документ: **`docs/Roadmap-flow-стриминга.md`** — работай по нему, майлстоунами Д0→Д8. (Прежний план, `docs/Evolution-TZ.md`, майлстоуны Э0→Э9, — архивирован и удалён из репозитория; Roadmap-flow-стриминга.md его сменяет.)

## Стратегия

Эволюция, **не переписывание**. На каждом шаге приложение остаётся рабочим. Голос через LiveKit SFU остаётся (это часть плана). Заменяется транспорт и раздача **видео**: стример шлёт поток на сервер (vrelay), сервер — постоянный медиаузел, раздаёт зрителям и транскодирует рендишны по требованию.

## Жёсткие инварианты (не нарушать)

1. **Сервер (vrelay) — постоянный медиаузел.** Нативный вещатель шлёт один поток ingest на серверный relay-узел (vrelay); vrelay раздаёт зрителям напрямую и держит **транскод рендишнов по требованию** (1080/720/480/360) отдельными ffmpeg-процессами — `source` остаётся passthrough (без транскода), рендишны кодируются под запрос конкретного качества. Это плановая архитектура, не фолбэк: TURN-relay и headless vrelay были осознанными фолбэками P2P-дерева в прежнем плане (Э-серия) — теперь сервер в тракте всегда, по умолчанию.
2. **Два параллельных источника вещания.** Нативный клиент (Tauri) вещает через сервер-первый relay-путь (H.264, см. п.4). Браузер вещает по **старому пути через LiveKit SFU** (`engine.share()`, VP8) — оставлено намеренно ради совместимости веба, не временный шаг и не к удалению. Зритель при коннекте **динамически** определяет транспорт конкретного стрима (пришёл ли он через tree-сигналинг или через LiveKit-комнату) и подключается соответствующим `VideoTransport` — выбор идёт по факту вещания, не билд-флагом (`VITE_VIDEO_TRANSPORT` не используется для этого).
3. **Браузерный зритель — строго лист дерева**: не ретранслирует никому и ни в каком виде (без исключений). Ретрансляция — только нативные зрители (passthrough) и сервер (vrelay, passthrough/транскод рендишнов). (Старое LiveKit-вещание из браузера, п.2, дерева не касается — это отдельный параллельный путь.)
4. **Кодек видео — H.264** для relay-пути (нативный вещатель ↔ сервер ↔ любой зритель), low-latency, без B-кадров (совместимость натив↔браузер↔сервер). Старый LiveKit-путь (веб-вещание) остаётся на VP8 — исторический кодек этого пути, не мигрируется.
5. **Микрофон не примешивается к стриму.** Голос — только через голосовой канал (SFU); чтобы говорить, надо быть в голосовом. Стрим = видео + звук игры/системы.
6. **Захват аудио не должен содержать вывод самого RelayApp** (голоса, чужие стримы, служебные звуки) — иначе эхо/петля. Нативно: WASAPI process loopback INCLUDE (только игра) или EXCLUDE (всё, кроме нашего процесса).
7. **Кодирование/масштабирование исходного видео — на стороне вещателя** (2K→1080p = рендер сцены в выходном разрешении на GPU); вещатель всегда шлёт `source` без транскода. **Сервер (vrelay) транскодирует рендишны** (1080/720/480/360) ffmpeg-процессами по требованию зрителя — осознанная часть архитектуры, не нарушение. Нативные зрители-ретрансляторы по-прежнему форвардят RTP как есть (passthrough); браузер не ретранслирует (см. п.3).
8. **VST/CLAP-обработка голоса — только в нативе**; в вебе fallback на Web Audio (EQ/компрессор/гейт, без плагинов).
9. Задержка видео **< 2 с** (цель сервер-первой архитектуры — прямая раздача с сервера, минимальная глубина дерева); голос ≤ 250 мс.
10. Несколько стримов в комнате = несколько деревьев (ключ `streamId`; при вводе рендишнов по требованию ключ дерева станет составным `streamId::rendition` — см. `docs/Roadmap-flow-стриминга.md`, майлстоун Д3).

# Подсистемы и уроки (накоплено по сессиям)

Архитектурные факты и грабли, которые НЕ выводятся из кода за минуту. Перед правкой соответствующей области — прочитать.

## Голос / аудио (engine.ts)

- **Изоляция звука по каналам:** подписка на mic пира только если `inVoice && !deafened && currentVc && attributes.vc===currentVc`. Единая точка — `reconcilePeerAudio` (+ `onRemotePub` для TrackPublished). `reconcileAllAudio` перебирает всех.
- **Фулл-мут (deafen) = ОТПИСКА, не громкость.** Раньше глушили `setVolume(0)`, но подписка оставалась → пир размутился → `ParticipantAttributesChanged` → resubscribe без re-apply громкости → звук проходил мимо мута. Теперь `!deafened` в `want`, `toggleDeaf` всегда зовёт `reconcileAllAudio` (deafen→отписка, undeafen→переподписка). Нет трека = гарантированная тишина. `leaveVoice` сразу сносит mic-аудиоэлементы (не screenshare — стрим смотрится и без голосового; отличаем по `screenAudioEls`).
- **Ростер голосового — по vc-АТРИБУТУ, не по mic-публикации.** При `autoSubscribe:false` mic-публикация «уже присутствовавших» пиров локально не приезжает → `isInVoice` давал false → пир пропадал из канала, хотя сам и сервер видят его в голосовом (симптом «я его не вижу, а он меня — да»). `voiceChannelOf` (attributes.vc) доставляется и для старых пиров. Fallback — серверный `voiceHint` из /presence.
- **setAttributes МЕРЖИТ** (не заменяет) — `{deaf}` не затирает `vc`. Но публикация может тихо провалиться (гонка оптимистичного входа / rate-limit / таймаут 5с): `setAttributes` ждёт серверного эха и обновляет `localParticipant.attributes` только по подтверждению → при неудаче реджект, атрибут пуст. Симптом «юзер невидим другим, появился через время (на реконнекте)». Лечит `selfHealVc` в 3с-таймере: пере-заявляет vc/deaf, ЕСЛИ опубликованный (server-confirmed) атрибут ≠ текущему состоянию. Двунаправленно (в войсе vc=currentVc; вне — vc='').
- **Индикатор качества связи:** `connQuality` читаем НАПРЯМУЮ из `localParticipant.connectionQuality` в conn-полле (2.5с), не ждём событие `ConnectionQualityChanged` (оно только при СМЕНЕ качества → при стабильной связи метка залипала на «соединение…»). Фолбэк из пинга, если LiveKit ещё отдаёт unknown.
- **Мультисессия:** LiveKit identity = `username#nonce`, наружу — `baseUid()` (base username). Своя другая сессия не подписывается (эхо). Одна голосовая на аккаунт — `vclaim` (reliable data + переотправка на Reconnected + tie-break гонки по session-id).
- **reliable data** (`dataSend`) для чата/vclaim/clear — терять нельзя. typing/emote/watch — эфемерны.
- **Шумоподавление — RNNoise (`denoise.ts`), self-host, three-mode** (`nsMode: 'off'|'basic'|'rnnoise'`, дефолт `rnnoise` для всех). Библиотека `@sapphi-red/web-noise-suppressor`, изолирована за `createDenoiseNode()`/`destroyDenoiseNode()` — любая ошибка загрузки WASM/AudioWorklet возвращает `null`, вызывающий фолбэкает на прямое соединение графа (голос не должен падать из-за шумодава). Выбран вместо DTLN/Krisp: Krisp требует LiveKit Cloud (мы self-host SFU), DTLN хочет cross-origin isolation (COOP/COEP) — глобальные заголовки, риск задеть аплоады/SW/натив-webview; RNNoise — ~1% CPU, без заголовков.
  - **Граф**: `устройство → [RNNoise?] → gain(мут/микс) → LiveKit`, с ОТДЕЛЬНЫМ отводом на VAD/метр **до** гейта (`micVadDest`), не после `micGain` — иначе анализатор видит уже замолченный (gain=0) сигнал и VAD-гейт залипает закрытым.
  - **`channelCount:1` в `micCapture()` обязателен.** `RnnoiseWorkletNode` сконструирована на `maxChannels:1`, а реальные микрофоны часто отдают gUM-поток 2-канальным по умолчанию (даже физически моно-капсюль) — рассинхрон каналов молча маскирует эффект шумодава в RMS/метре (обработан только канал 0, необработанный канал 1 доминирует в анализе). Синтетические тесты с моно WAV-файлом (`--use-file-for-fake-audio-capture`) этот баг не ловят — нужен реальный мультиканальный микрофон.
  - **`ctx.audioWorklet.addModule()` — per-AudioContext, дедуп через `WeakMap`** в `denoise.ts` (`ensureWorkletModule`): повторный `addModule` с тем же URL на ТОМ ЖЕ контексте кидает "already registered" (воркет-скрипт перевыполняется в той же глобальной области, `registerProcessor` дублируется). Актуально при переиспользовании контекста между вызовами превью-метра.
  - **Превью-метр настроек (вне звонка) — свой независимый путь** (`startLevelMeter`/`stopLevelMeter`/`restartLevelMeter`), не связан с in-call графом. Раньше смена `nsMode` в открытых настройках НЕ перезапускала уже бегущий метр (пришлось добавить `restartLevelMeter()`) — иначе пользователь видит старое поведение до выхода из настроек.
  - **Гонка «стоп во время загрузки WASM»**: между `await createDenoiseNode(ctx)` и продолжением сборки графа — снимок `ctx`/`stream` до await, сверка после (`this.micActx !== ctx` / `this.levelStream !== stream`) — конкурентный `stopMic()`/`stopLevelMeter()` (watchdog реконнекта мика) мог обнулить поля контекста, пока денойзер ещё грузится.
  - **`RnnoiseWorkletNode` пишет реальный сигнал ТОЛЬКО в канал 0 своего выхода** (канал 1 — тишина), независимо от `channelCount:1` на входе. Без явного `ChannelSplitterNode` после денойзера (`.connect(next)` без output-индекса = output 0, чистый канал 0) следующий узел графа видит «2-канальный» сигнал с тишиной справа → апмикс на publish отдаёт звук только в ЛЕВОЕ ухо собеседникам (в превью-метре — не рвёт стерео, а вдвое занижает уровень, т.к. локальный анализ downmix'ит 0.5×(L+0)). Поймано и подтверждено количественно двухпирным тестом (A вещает, B слушает, замер L/R принятого трека через ChannelSplitter+AnalyserNode на стороне B) — синтетический OfflineAudioContext-тест с тестовым тоном тут ОБМАНЫВАЕТ (RNNoise корректно давит нечеловеческий сигнал в тишину на ОБОИХ каналах, левое/правое неразличимо) — нужен живой граф с речеподобным сигналом или реальный звонок.
  - **Тестирование с РЕАЛЬНЫМ микрофоном пользователя** (не синтетический fake-audio): Chrome с `--remote-debugging-port=9222 --user-data-dir=<temp>` (отдельный профиль — иначе новый `chrome.exe` просто откроет вкладку в уже запущенном процессе, флаг не подхватится) → Playwright `chromium.connectOverCDP('http://127.0.0.1:9222')` находит существующую страницу по URL и читает/кликает по ней вживую, с реальным звуком пользователя. НЕ звать `browser.close()` на CDP-подключении к чужому окну — оно реально закроет браузер пользователя.

## Уведомления (многослойно)

- **Локальные** (`notify.ts`): натив — кастомное окно-карточка (Tauri WebviewWindow, см. ниже) с фолбэком на OS-toast; веб — `reg.showNotification` (SW) с фолбэком `new Notification`. Гейт фокуса ТОЛЬКО для mention (Discord-стиль); стрим/обновление — всегда. Общий тег `<kind>:<serverId>` в local И push → ОС схлопывает в один баннер (без дублей). Автозапрос разрешения при старте (opt-out флаг `notifOptOut`); iOS промпт нужен по ЖЕСТУ (тумблер), авто-запрос там игнорится.
- **Web Push (VAPID)** (`push.ts` + сервер): фон/закрыто, единственный путь для iOS PWA (в фоне JS заморожен). Ключи авто-генерятся в `data/vapid.json` (стабильно). `urgency:'high'` + TTL — КРИТИЧНО (без high FCM/APNs батчат → «не сразу»). Room-gate УБРАН (свёрнутая мобилка ещё ~15-20с «в комнате» → окно пропуска). SW ВСЕГДА `showNotification` в `waitUntil` (iOS: тихий push → Safari снимает подписку). Подписка перепривязывается к аккаунту в `afterAuth` (минуя opt-out гейт); logout отписывает под старым токеном.
- **Глобальный notify-WS** (`/ws`, `notifyws.ts`): для уведомлений о НЕ подключённом сервере (LiveKit-комната поднята только для текущего; натив web-push не получает). Сервер на mention/stream шлёт `notifyUser(uid)` напрямую по user_id → его ws-сокеты. Клиент дедупит по `connectedServerId` (текущий сервер обслуживает live LiveKit-путь).
- **Кастомная нативная карточка** (`notif.html` + `notif-window.ts`, Vite multi-page): своё окно-оверлей (как Discord/Steam), прозрачное/без рамки/поверх всех/non-focus, правый нижний угол. Позиция — `currentMonitor().workArea` в ГЛОБАЛЬНЫХ логических px (screen.availWidth не учитывает офсет монитора → Tauri центрировал окно). Capabilities: главному — `core:webview:allow-create-webview-window`+`core:window:allow-close`; окну `notif` — `core:default`+close/set-focus/unminimize/show. Клик по карточке → фокус главного окна; `focusable:false` (не крадёт фокус).
- **Таскбар-бейдж:** `navigator.setAppBadge` (установленная PWA). Сумма непрочитанного + флаг обновления.

## Непрочитанное (unread)

- Сервер: таблица `read_state(user,server,last_read)`. `/me` отдаёт `unread`+`lastRead` per server; `POST /servers/:id/read`; лёгкий `GET /unread` (без LiveKit). unread = чат-сообщения новее last_read и НЕ свои. **Первое обращение лениво сеет** last_read=последнему (иначе на раскатке весь бэклог = непрочитан).
- Клиент (store): `unread`/`lastRead`-мапы. Активный (connected) сервер клиент ведёт сам (`bumpUnread` на новое не-своё пока не читаем, `markRead` внизу чата); остальные — поллинг `/unread` 30с (mergeUnread не трогает connected). Рейл-бейдж на иконке сервера. В чате — дивайдер «Новые сообщения» перед первым непрочитанным + `initialTopMostItemIndex` позиционирует туда, jump-кнопка (⌄) со счётчиком. Скролл в конец на отправку/фокус инпута.

## Чат (надёжность)

- **Дедуп mergeRecent** (реконнект-догрузка): по sid И по сигнатуре (автор+текст+картинка) — live-эхо от onData не имеет sid, иначе refetchChat дублировал. Совпавшему live-сообщению «усыновляем» серверный sid. Свои НЕ пропускаем безусловно (мультисессия: своё с другого устройства могло не дойти).
- **Идемпотентность POST**: клиент шлёт `client_key` (переживает retry) → partial-unique индекс `(server_id,user_id,client_key)` + `INSERT OR IGNORE`. `info.changes===0` = дубль (retry после потери ответа) → не пушим повторно.

## Оркестрация соединения (store.ts)

- **Эпоха соединения (`connEpoch`)**: инкрементится на connectServer/exitServer/logout. Фоновый connect-IIFE и member-poll сверяют эпоху перед записью — иначе протухший хвост прошлого сервера рвёт живую комнату нового / пишет чужой состав. Убран безусловный disconnect по устаревшему `connectedServerId` (убивал живую комнату). Провал фонового connect сбрасывает `connectedServerId` (иначе повторный клик → showConnectedServer без реконнекта → realtime мёртв до F5). `goHome` НЕ рвёт соединение.
- **fetch-таймаут 15с** (AbortController) в `api.req` — мёртвый TCP иначе оставлял вечное «отправляется».

## Видео-дерево / discovery (treeVideo.ts)

- **Tree-liveness самолечится ре-`hello`** (15с): `liveStreams` мутируют только discovery-события → потерянный `stream-live` (полуоткрытый WS) висел до F5. onHello на сервере идемпотентен (переотдаёт бэклог), клиентский `fresh`-гард без дублей. LiveKit-стримы самолечатся 3с-таймером (живой опрос комнаты).
- `openDiscovery` при throw конструктора WS планирует ретрай (раньше discovery умирал навсегда). 2с-реконсиляр НЕ сносит активные watch (окно 4с). Закрытие прежнего сокета + гард реконнекта (только текущий сокет).
- **Грид до 4 стримов разом** (`WATCH_MAX` в engine.ts). Rust держит watch-слот НА КАЖДЫЙ стрим (`WatchState = Mutex<HashMap<streamId, RelayHandle>>`), `stopNativeWatch(streamId)`/`watch_answer`/`watch_ice`/`watch_reparent` адресные по streamId (relay-core уже тегирует webview-события `streamId`). Кап — единая точка `engine.watch` (reject+тост на 5-м), покрывает и веб (свой tree-WS/PC на стрим), и натив. Раньше был ОДИН слот (`Mutex<Option>`, `stopNativeWatch()` глобален), `engine.watch` закрывал прочие. Грид-верстка — `#grid.n3/.n4` (2×2) в styles.css, `Stage` выбирает класс по числу плиток.

## Server-first + серверный транскод (Roadmap-flow-стриминга, Д-серия)

Дефолт — **«стример → сервер (vrelay) → зрители»** (Д8: `TREE_SERVER_FIRST` дефолт-on, `!== '0'`; откат = `0`). vrelay = постоянный медиаузел (pinned прямой ребёнок корня в source-дереве; **корень** в рендишн-деревьях). Активен ТОЛЬКО для стримов с `serverIngest:true` (нативный вещатель); стримы старых бандлов (без поля) идут по **LEGACY**-фолбэку (дренаж, `VIRTUAL_COST`-штраф, idle-exit) — эти ветки помечены маркером `// LEGACY (снос после раскатки натив-клиента ≥ vX)`.

- **Ключ дерева — составной `streamId::rendition`** (`treeKey`/`parseTreeKey`), но живёт ТОЛЬКО внутри сервера. Все исходящие клиенту сообщения (`assign-*`, `tree-info`, `tree-topology`, `stream-*`, `set-bitrate`, `drop-peer`) несут **базовый** streamId — иначе теряется топология у натива (`topoCb` сверяет базовый id из watch). Discovery агрегирует по базовому id.
- **Рендишн-корень — `role:'broadcaster'`, но `virtual:false`** (иначе self-loop в `ensureVirtualAttached`); его `native:true` (иначе `capacityOf` даёт 0 — гард `!node.native`). Его `stream-live`/`stream-end` гейтятся `rendition==='source'`: иначе `stream-end` рендишна **гасит живой source-стрим** у зрителей, а `stream-live` плодит дубль с мусорным identity (`vrelay-480`).
- **`register_default_codecs` у probe-приёмника (probe.rs) обязателен** ради **transport-cc** — этим probe и отличается от relay, НЕ отсутствием nack. У peer.rs (вещатель) `rtcp_feedback:vec![]` в register_codec — **no-op**: `register_default_interceptors`→`configure_nack` ретроактивно дописывает nack/pli ко всем видео-кодекам (проверено по webrtc 0.17.1), NACK на H.264 работает. У **relay.rs** сборка Registry теперь РУЧНАЯ (ради ускоренного Generator), поэтому nack-фидбэк на H.264 задан ЯВНО (`RTCPFeedback nack`+`nack pli`) — без configure_nack он бы не активировался; Responder кэш 8192, инлайн-повтор без RTX (Chrome-зритель принимает по RFC4585). Send-side GCC-BWE вещателя без transport-cc не разгоняется. Замер upload вещателя — из **webview** (Chromium GCC), не из webrtc-rs.
- **NACK есть, но «не успевает» — вот источник остаточных фризов, не «его нет».** Разбор diag penis14 (2026-07-10): случайная потеря последней мили (3.4% вечером у RU-зрителя) восстанавливается ретрансмитом (в ingest-логе `переупорядочено`≈`разрывам` = ретрансмиты приходят), но опоздавший за приёмный буфер пакет декодер выбрасывает → фриз до keyframe. Рычаги (сделаны): приёмный буфер зрителя (`JITTER_TARGET_MS` в treeVideo.ts = 500мс — 2 NACK-цикла); `GAP_HEAL_GRACE` (relay.rs = 250мс — не дёргать IDR, пока NACK не провалился); `NACK_GENERATOR_INTERVAL` (relay.rs = 50мс вместо дефолтных 100 — вдвое быстрее детект пропуска, через ручной Registry). Дальше (НЕ сделано, отдельные проекты): FEC (ulpfec/flexfec — webrtc-rs не имеет), реальный send-side BWE (`webrtc-ice` хардкодит `available_outgoing_bitrate=0`). Пейсер фанаута (`PACE_*` в relay.rs) — против line-rate keyframe-микробёрста, второстепенно (форензика: `lostPerGap≈1`, потери НЕ бёрстовые). Generator-интервал в native-зрителе действует только после пересборки натива; JITTER_TARGET_MS (webview) — сразу с веб-деплоем.
- **`webrtc-ice 0.17.1` хардкодит `available_outgoing_bitrate: 0.0`** (congestion control не реализован) → BWE нативного ЗРИТЕЛЯ тоже берётся из Д5-probe-кэша (webview), не из webrtc-rs. `availableOutgoing=0` на сервере = «не измерен» → консервативная ёмкость 1 (не раздуваем супер-сидеров на фейке).
- **Нативный webview-PC — лупбек** (Rust↔webview, `packetsLost≈0`): клиентский frame-drops-детектор для натива структурно молчит. Отбраковку плохого родителя у натива делает **сервер** (`frameDropReparent` по upstream `linkLoss` из `toChild` родителя). `framesDropped` в Chromium включает рендерные дропы слабого ПК → **второй сигнал** (`packetsLost`/`linkLoss`) обязателен, иначе ложные миграции.
- **`moveNodeToRendition` сбрасывает `reparentCooldownUntil`** (свежее дерево) → cooldown против болтанки ABR ставится **ПОСЛЕ** move, иначе следующий тик двигает узел снова.
- **Cooldown НЕ защищает от орбиты «корень ↔ vrelay».** Он запрещает двигать узел часто, но не мешает через 10с вернуть его ТУДА ЖЕ, а корень со свободными слотами всегда «лучший» по `scoreParent` (глубина 0). Живой кейс: вещатель с `maxChildren=5` (слайдер «прямых подключений»), зритель под vrelay → `request-reparent` от клиента → `pickParent` исключает текущего родителя, единственный кандидат корень → домашний аплинк не тянет вторую копию потока, loss 18% → `frameDropReparent` уводит обратно → круг; у зрителя 7 watch-сессий за 2 минуты. Лечит `badParents` (карантин `BAD_PARENT_TTL_MS`); смягчение карантина — **только для сироты** (`forNode.parent === null`), иначе «запасным» окажется как раз забракованный. Регрессия — `apps/server/tree-flap-test.js`.
- **Прямые дети вещателя делят его аплинк.** Один энкод, но по копии потока каждому: 2 ребёнка при 4.5 Мбит = 9 Мбит вверх. На домашнем канале это потери у ВСЕХ детей разом, включая vrelay, и ABR валит битрейт всему дереву. Прямой P2P от вещателя выигрывает только внутри LAN.
- **`PRESET_TABLE` дублируется** `presets.ts` (источник истины, ESM-бандл) ↔ `tree.js` (копия для валидации `RENDITION_BITRATE`, CommonJS — общего рантайма нет). Правишь значения — меняй ОБА; в `tree.js` есть warn-валидация при старте.
- **ffmpeg-транскод** (`transcode.rs`): `kill_on_drop(true)` + явный kill (иначе зомби на VPS); `ActiveGuard` слота захватывается **внутрь** `async move`, иначе дропается сразу. GOP 60 (2с), `-bf 0`, CBR+HRD (`nal-hrd=cbr`), scale без апскейла; Opus — passthrough (через ffmpeg НЕ идёт). Канал rendition-команд — control-WS агента (рядом с `vrelay-ingest`), не per-stream сессия.
- **`request-keyframe` из рендишн-контекста легитимен ТОЛЬКО из `ensureRendition`** (ffmpeg без IDR не декодирует старт). PLI из рендишн-дерева НЕ уходит нативному вещателю (`onRequestKeyframe` шлёт корню ЭТОГО дерева — для рендишна это ffmpeg с фиксированным GOP) — нет IDR-шторма через деревья.
- **coturn `external-ip`** — не хардкод: флаг `--external-ip=$TURN_EXTERNAL_IP` из docker-compose (`${VAR:+...}` — пусто → флага нет, coturn не падает). Прод обязан задать `TURN_EXTERNAL_IP` в `.env`.
- **`dev-rendition` DEV-триггер удалён в Д8** (Д4 сделал настоящий реестр рендишнов).
- **Медиа-стек вынесен на отдельный VPS** (`deploy/vrelay-remote/`), в корневом compose `vrelay` и `coturn` живут под профилем `local-media` и по умолчанию НЕ поднимаются. Откат — `COMPOSE_PROFILES=local-media` в прод-`.env`. Почему так: **vrelay — WS-КЛИЕНТ, не сервер** (`main.rs:41` читает `TREE_WS_URL`, коннектится исходящим, портов не слушает), значит на удалённом хосте не нужны ни Caddy, ни TLS; `tokio-tungstenite` собран с `rustls-tls-webpki-roots`, поэтому `wss://` работает из коробки. **coturn обязан ехать вместе с vrelay**: единственный его потребитель — дерево (`iceServersFor` зовётся только из `welcome`), LiveKit к нему не обращается (свой ICE, `7881/7882`); оставить его на проде = зритель за симметричным NAT идёт `зритель → прод-coturn → vrelay`, лишний хоп через машину, от которой уехали.
- **Двух vrelay-агентов одновременно быть не должно.** `findVrelayAgent()` берёт ПЕРВОГО `isVrelayAgent` из `peers` (порядок вставки Map) — при двух подключённых выиграет тот, кто пришёл раньше. Env-переключателя нет; «активен тот, чей контейнер запущен».
- **Свой STUN** — тот же coturn: Binding-запросы авторизации не требуют (`use-auth-secret` гейтит только TURN-аллокации). `tree.js` давно принимал opt `stunServers`, но `index.js` его не передавал — теперь читает `STUN_URLS` (свой первым, Google последним: coturn лёг, а vrelay жив → srflx всё равно добывается). В копии `turnserver.conf` для удалённого хоста снят `verbose` — иначе каждый Binding пишет строку, и 100 МБ json-file выкручиваются за дни.
- **`natDetect.ts`: ОДИН `RTCPeerConnection` с обоими STUN-серверами.** Сравнивать надо внешние порты ОДНОГО локального сокета (общий `relatedPort`), увиденные разными серверами. Два независимых PC (как было) дают два разных локальных порта, а ЛЮБОЙ NAT отображает их на разные внешние — проба возвращала `symNat=true` практически для каждого веб-зрителя. Браузер схлопывает одинаковые srflx-кандидаты, поэтому конусный NAT даёт ОДИН кандидат на оба сервера, симметричный — два с разными портами. Серверы обязаны быть на РАЗНЫХ ХОСТАХ (мапинг per-destination-IP): список из `welcome` через `stunUrlsByHost` (свой coturn + Google), проба перезапускается на welcome.
- **Симметричный NAT НЕ мешает СМОТРЕТЬ** в server-first: у vrelay публичный IP, зритель шлёт наружу первым, узел отвечает на peer-reflexive адрес. Флаг влияет только на `capacityOf` (может ли узел быть родителем) и при выключенном TURN обнуляет ёмкость. STUN симметричный NAT не лечит по определению — только TURN.
- **`TURN_LISTEN_IP` на облачном хосте обязателен.** У DO-дроплета кроме публичного IPv4 на `eth0` висит ещё `10.19.0.5`, на `eth1` — VPC `10.114.0.2`, плюс `docker0`. Без привязки coturn раздаёт relay-кандидаты с приватных адресов, и зритель жжёт на них ICE-таймауты.

## Захват экрана: CPU-путь и приоритеты (broadcast/capture.rs, prio.rs)

- **Весь видеотракт вещателя — на CPU**, вопреки инварианту 7. WGC отдаёт кадр как `ID3D11Texture2D`, но `frame.buffer()` делает GPU→CPU readback (внутри крейта — staging-текстура + `CopyResource` + `Map` на КАЖДЫЙ вызов), дальше скейл и BGRA→NV12 считаются скалярно. Хардварный MFT при этом работает на system-memory: `is_d3d_aware`-энкодеры (`encoder.rs`) намеренно отбрасываются. Zero-copy GPU-путь (D3D11 VideoProcessor + `IMFDXGIDeviceManager` + `MFT_MESSAGE_SET_D3D_MANAGER`) не реализован — при этом всё, что нужно, у `Frame` уже есть (`as_raw_texture`/`device`/`device_context`).
- **MMCSS обязателен, иначе игра вытесняет захват.** Игра бежит с MMCSS-задачей `Games`; наши `std::thread::spawn` шли с NORMAL и голодали первыми при 100% CPU: WGC ронял презенты (падал `raw` в логе), энкодер не успевал забрать кадр из `bounded(2)` → `capture_drops`. `prio::ensure_thread_mmcss` регистрирует поток (идемпотентно, thread-local, revert на TLS-деструкторе — `AvRevertMmThreadCharacteristics` нельзя звать с чужого потока). Применён к capture-сессии, encoder-потоку, audio-потокам и **воркерам rayon-пула** — без последнего смысл теряется, тяжёлая часть кадра считается именно там. `THREAD_PRIORITY_ABOVE_NORMAL`, не `TIME_CRITICAL` (конверсия тяжёлая, заголодает систему).
- **Колбэк `on_frame_arrived` — на том же потоке, что `Capture::start`** (frame pool у windows-capture 2.0 создан не free-threaded). Поэтому регистрации MMCSS в потоке сессии достаточно. Долгий колбэк тормозит саму WGC-сессию: «мы не успеваем» и «источник мало презентит» дают ОДИНАКОВО низкий `raw` — различает только метрика времени колбэка (`timing:` в stats-тике, `SharedStats::cb_ns`).
- **Никаких per-frame `thread::scope`/аллокаций.** Конверсия крутится на статичном `CONVERT_POOL` (rayon, `OnceLock`), NV12-буферы ходят по кругу через `BufferPool` (возвратный `crossbeam`-канал; энкодер обязан `put` после `encode`, включая ветки-`continue`). Раньше было до 12 spawn+join и ~3 МБ alloc НА КАДР.
- **`bgra_to_nv12_luts` обязан ловить кадр 0×0** (свёрнутое окно): `chunks_mut(0)` паникует. Тест `parallel_conversion_matches_reference` сверяет параллельный путь с эталонной петлёй — смещение `row0` у бэнда единственное место, где кадр «поедет полосами» при том же fps/битрейте.
- **Адаптации к CPU нет.** `QualityLadder` реагирует только на сетевой `set-bitrate` и выключена в server-first/preset-режимах (`ladder_enabled`). При перегрузке CPU качество не деградирует — кадры просто дропаются на `bounded(2)`.
- **Идеальные цифры захвата НЕ значат, что у зрителей всё хорошо.** Реальный случай: `accepted=30.0 raw=55.0`, ноль `capture_drops`, `cb` 3.7мс из 33мс бюджета, `encode` 1.0мс — а зрители (все разом) видели секундные фризы. Если картинка подвисает у НЕСКОЛЬКИХ зрителей одновременно, виноват общий участок ПОСЛЕ энкодера: аплинк вещатель→vrelay, coturn или сервер. Улики — строка `net:` (loss/rtt по линку, PLI, IDR) и `freezeCount` у зрителя, а не `timing:`.
- Спин ожидания кредита async-MFT (`encoder.rs`) — шаг 500мкс, не 1мкс: кредита нет ровно под перегрузкой, и прежний спин выжигал ядро, ухудшая то, что пережидал. Блокирующий `GetEvent(0)` честнее, но не имеет таймаута → залипший MFT подвесил бы поток без пути к остановке.
- **GOP задаётся `MF_MT_MAX_KEYFRAME_SPACING` на ВЫХОДНОМ ТИПЕ, до `SetOutputType`.** Аппаратный MFT читает конфигурацию в момент `SetOutputType` и молча игнорирует `CODECAPI_AVEncMPVGOPSize`, выставленный после — энкодер оставался на дефолтном GOP=fps, то есть IDR раз в СЕКУНДУ вместо раза в 4с. Улика в net-логе вещателя: `IDR +2` за 2с-окно при `PLI +0` (значит IDR никто не просил, их печёт сам энкодер). Симптом у зрителя — фризы 200-600 мс **без единой потери пакета**: 1080p-IDR при 4.5 Мбит это ~200-300 КБ, ~170 пакетов залпом, полсекунды канала. `ICodecAPI`-вызов оставлен вторым поясом для софтверных MFT.

## Диагностика стрима (diag.rs / diag.ts / POST /api/diag/session)

- **`PLI +N` в `net:`-строке считает ТОЛЬКО RTCP-PLI от прямых детей** (`peer.rs`). Запрос кейфрейма из глубины дерева приходит СИГНАЛИНГОМ (`TreeEvent::RequestKeyframe` через vrelay) и раньше форсил IDR, не трогая счётчик — `PLI +0 | IDR +2` читалось как «IDR никто не просил». Теперь сигналинг-путь тоже инкрементит `pli_count`. При чтении старых логов помнить об этой слепоте.

- **Две строки в логе вещателя раз в 2с.** `timing:` — секции горячего пути (cb = readback + convert, encode, write, drops). `net:` — loss/rtt по каждому прямому ребёнку, факт/цель битрейта, PLI и IDR за окно. До этого `link_stats` уходили только серверу, а PLI писался на `debug` при `LevelFilter::Info` — то есть молчал. Отличить «вещатель не успевает» от «сеть сыпется» по файлу лога было нечем.
- **Логи сдаются на сервер по окончании сессии.** `diag.rs` — кольцевой буфер (20k строк / 2 МБ), подключён как `TargetKind::Dispatch` рядом со Stdout/LogDir, поэтому ловит и webrtc-rs (ICE/TURN-ошибки). Фронт забирает его через `diag_take_log` и постит вместе со своими семплами — HTTP делает веб-сторона, у неё уже есть session-JWT.
- **Зритель шлёт `freezeCount`/`totalFreezesDuration`** из `inbound-rtp` — прямая мера симптома «картинка подвисает», а не косвенные потери. Семпл раз в 2с. У нативного зрителя PC — лупбек webview↔Rust (`packetsLost≈0`), но `freezeCount` там честный: он про то, что видит глаз.
- **`express.json` ставится ДО матчинга путей**, поэтому глобальный `limit:'32kb'` резал бы диаг 413-м. Парсер выбирается по `req.path` (см. `DIAG_MAX_BODY`). Капы: тело 2 МБ, `slice` в роуте, каталог `data/diag` — 400 файлов / 100 МБ, `pruneDiag()` после каждой записи.
- **Потери выгрузки:** вкладку закрыли → `pagehide` + `fetch(keepalive)` (лимит тела 64 КБ, поэтому только семплы, без лога). POST не прошёл → очередь в localStorage, дослать в `afterAuth` (`flushPendingDiag`). 4xx не ретраим — забили бы очередь. Полный краш процесса выгрузку теряет; локальный файл лога при этом цел.
- Сессия сбрасывается (`diag::reset`) на `start_broadcast`/`start_watch`. Rust-буфер один на процесс: если натив одновременно вещает и смотрит, строки заберёт та сессия, что закончилась первой.
- **Серверная сторона логировала только топологию.** `tree.js` уже принимал `linkLoss`/`linkRtt` по КАЖДОМУ линку (vrelay репортит по каждому зрителю, вещатель — по прямым детям), клал в EWMA и использовал для ABR — но не печатал: «у всех зрителей фризы» на сервере выглядело как тишина. Теперь `treeHealth()` в abr-тике печатает плохие линки (гейт `HEALTH_LOSS`/`HEALTH_RTT`, baseline раз в минуту).
- **`IngestMonitor` (relay.rs) — решающая улика.** Разрыв `sequence_number` на ВХОДЕ relay-узла: пропуски есть → теряет линк от родителя (для vrelay это аплинк вещателя); пропусков нет, а зрители фризят → виноват узел или линк вниз. `wrapping_sub` + порог `0x8000` отделяет потерю от переупорядочивания (тесты в `relay.rs`). Раньше `write_rtp`-ошибки шли в `log::debug!` (при `RUST_LOG=info` — молча), а обрыв upstream был голым `break` без строки.
- **Логи Docker обязаны быть ограничены.** `json-file` по умолчанию пишет без предела, контейнеры живут месяцами. Якорь `x-logging` в `docker-compose.yml`: 100 МБ × 2 на сервис, vrelay 200 МБ × 3 (он самый многословный) ≈ 1.5 ГБ + `data/diag` 100 МБ.
- **Читать логи — только через `scripts/diag.mjs`** (`npm run diag -- ...`). Сырая сессия вещателя = до 2 МБ, глазами/моделью не читается. `list` / `pull --stream=<id>` / `report .diag/*.json`: отчёт сводит вещателя и зрителей на общую шкалу epoch-ms (окна по 2с), печатает ТОЛЬКО окна с фризами/потерями/дропами (±1 окно контекста), схлопывает повторяющиеся ICE-строки и выносит вердикт «вещатель не успевал» / «линк сыпется» / «смотри сервер». Авторизация: `RELAY_TOKEN` либо `RELAY_USER`+`RELAY_PASS`; нужен админ.
- **Вердикт считается долями, не абсолютами.** Прежний гейт `drops > 0 || cbMax > 30` обвинял захват за 55 дропов на 316 окон (10 минут) и увёл разбор не туда. Сейчас: `dropWindows/windows > 0.1` ИЛИ `cbMax > 1000/targetFps` (бюджет кадра, не константа 30) → вещатель; иначе `maxLoss`/`worstViewerLoss > 2%` → линк. `idrRate > 0.5/с` печатается отдельным предупреждением.
- **Строки диага двойного префикса**: `tauri-plugin-log` кладёт СВОЙ отформатированный префикс внутрь `record.args()`, поэтому строка выглядит как `<epoch> [INFO][target] [date][time][target][INFO] net: ...`. Парсер `diag.mjs` обязан искать `includes("net:")`, а не `startsWith`.

## Аплинк VPS — потолок раздачи (разбор прода 2026-07-09)

Симптом: у ВСЕХ зрителей разом секундные фризы, при идеальных цифрах захвата. Причина — исходящая полоса VPS, а не клиент.

- **Реальная устойчивая отдача 138.16.170.21 — 14–20 Мбит/с** (замер одним потоком на простаивающем сервере), при загрузке 68 Мбит/с. Прежние `VRELAY_OUT_MBPS=37` замерены 6 параллельными TCP-потоками: на шейпленном канале они съедают burst-кредит и врут вверх. Полосу мерить ОДНИМ потоком.
- Улики в ядре при переполнении: `Udp SndbufErrors` и `dropped` у qdisc растут, а `RX/TX dropped` у NIC — нули. То есть дропает наш стек на отдаче, не железо.
- **Порядок отказа читается по `[health]`**: сначала rtt одного зрителя прыгает (29 мс → 458 мс), через 8 с — потери 35–62% у всех четырёх, и только спустя ~16 с появляются разрывы на ingest vrelay. Ingest ломается ПОСЛЕДНИМ — потому что забитый egress-сокет бэкпрешурил `write_rtp`, тормозил read-петлю и переполнял внутренние буферы webrtc-rs. Отсюда развязка: read-петля пишет в `mpsc(256/100)`, отдельная writer-таска раздаёт, свои дропы считаются в `fanout_drops` — иначе собственная перегрузка неотличима от сетевых потерь.
- **Бюджет режет БИТРЕЙТ, не ёмкость** (`serverBudget` = `out × 0.7 / детей`, пол `ABR_FLOOR`). Урезать число детей нельзя: браузерный зритель не ретранслирует (инвариант 3), лишний стал бы сиротой без картинки. Пусть смотрят все, просто хуже. Тест — `apps/server/dfix-budget-test.js`.
- **ABR обязан работать и в пресет-режиме.** `BroadcastModal` слал `autoBitrate: false` при выбранном пресете → сервер не отправлял `set-bitrate`, и `bitrateTargetBps` стоял 4.5 Мбит все 630 с, включая окна с 46% потерь. Теперь `preset ? true : cfg.autoBitrate`. Пресет задаёт fps/разрешение, битрейт остаётся адаптивным.
- **Петля PLI→IDR→всплеск→потери→PLI.** При потерях IDR шли 0.99/с вместо ожидаемых 0.25/с — упирались в 1-секундный рейт-лимит. `onRequestKeyframe` удваивает `t.kfBackoffMs` на каждый форвард (кап `KF_FORWARD_MAX_MS=8000`), сбрасывает после 5 с тишины. **На джойне зрителя бэкофф сбрасывается принудительно**, иначе новый зритель ждал бы первый IDR до 8 с.

## Инфра / грабли

- **RTK-хук ломает `git commit -m @'...'`** (добавляет литерал `@`) — коммить через `git commit -F - <<'MSG'` heredoc.
- **Rust компилится локально** (`cargo check`/`cargo test --lib` в `apps/native/src-tauri`, toolchain стоит) — прежняя запись «нет toolchain» устарела. Сигнатуры Win32 всё равно быстрее сверять по vendored-исходникам (`~/.cargo/registry/src/*/windows-<ver>/src/Windows/Win32/...`), чем гадать; ACL-права Tauri — по `node_modules/@tauri-apps/api/*.d.ts` (неверное имя = ФЕЙЛ сборки). PROPERTYKEY — в `Foundation`, не PropertiesSystem. Cargo.lock в репозитории отстаёт от `Cargo.toml` — CI синкает его через `cargo fetch` (build-windows.yml), локальный `cargo check` тоже его дописывает.
- **Деплой**: push с правками `apps/web/**`/`apps/server/**` триггерит `deploy.yml` → пересоздание контейнеров РВЁТ живые relay-деревья (стримы падают). Натив (`capabilities/*.json`, `src/**`) → `build-windows.yml`, грузит бандл + capabilities вшиты в бинарь (веб-деплою не подчиняется).
- **`docker compose restart <сервис>` падает, если сервис вне активного профиля** — и валит весь деплой (`set -e`). Поэтому прежний `restart coturn` из `deploy.yml` снят вместе с уходом coturn под `local-media`. При этом `up -d` профиль-выключенные сервисы НЕ останавливает: уже запущенный контейнер продолжает жить, гасить руками.
- **Переливка `.env` через Windows-шелл приклеивает `\r`.** `SESSION_SECRET` длиной 65 вместо 64 = подпись JWT не сойдётся с прод-овой, агент получит отказ на `/tree`. Проверять `awk -F= '{print $1, length($2)}'`, чистить `sed -i 's/\r$//'`.
- **На чистом Ubuntu-дроплете `docker compose` может отсутствовать** (пакет `docker.io` ставит только CLI+демон): `docker: unknown command: docker compose`. Ставится `apt-get install -y docker-compose-v2`.
- **Переезд медиа-хоста (vrelay/coturn) — не атомарный, порядок обязателен.** `welcome` со списком ICE-серверов клиент получает РАЗ на сокет; живые PC не переконфигурируются, версии у списка нет — клиенты держат старый coturn в ICE, пока их tree-WS не пересоздастся (лишние ICE-таймауты в переходном окне, диаг 2026-07-10). Процедура: (1) поднять новый медиа-стек (`deploy/vrelay-remote/`, `docker compose up -d`); (2) обновить `TURN_URLS`/`STUN_URLS`/`TURN_EXTERNAL_IP` в прод-`.env`; (3) **рестарт прод-сервера** (рвёт все tree-WS → клиенты реконнектятся → свежий `welcome` с новыми ICE); (4) ТОЛЬКО потом гасить старый coturn. Гасить раньше рестарта = зрители жгут ICE на мёртвый TURN до реконнекта.
